# 개발팀 페르소나 시스템 보강 프롬프트 가이드

> **작성일**: 2026-03-29
> **목적**: 개발 완료 검증 실패, 오류 투성이 결과물, 진도 파악 불가 문제 해결
> **적용 대상**: 기존 78명 페르소나 시스템 (CLAUDE.md) 위에 덧씌우는 운영 규칙

---

## 현재 시스템 진단: 왜 "완료"인데 안 되는가

### 근본 원인 5가지

| 번호 | 문제 | 원인 | 결과 |
|------|------|------|------|
| 1 | 완료 확인 불가 | "코드 작성 = 완료"로 간주. 실제 실행 검증 단계 없음 | 보고는 완료, 실제는 미완성 |
| 2 | 접속하면 오류 | 각 팀이 자기 파트만 완료 보고. 통합 동작 검증 없음 | 부분은 OK, 전체는 깨짐 |
| 3 | QA/시뮬레이션 무력화 | QA가 "완성품"을 받아서 테스트. 이미 구조적 문제 내재 | 표면 테스트만 가능 |
| 4 | 개발 진도 불투명 | 중간 검증 없이 최종 보고만 존재 | 80% 완료라더니 실은 30% |
| 5 | 요청 방식 혼재 | 기획문서 vs 즉석 요청의 처리 경로가 동일 | 준비도에 따른 차별화 없음 |

### 핵심 처방: "보고 기반 완료" → "증거 기반 완료"

기존 시스템에서 빠진 것은 **"실제로 돌려보고 확인하는 단계"**입니다.
아래 두 프롬프트는 이 검증 체계를 강제로 삽입합니다.

# 프롬프트 A: 기획문서 첨부 기반 개발 요청

> **사용 시점**: PRD, 기획서, 화면설계서 등 문서를 첨부하여 개발을 요청할 때

---

```
@PM팀 기획문서 기반 개발 요청

## 첨부 문서
[기획문서/PRD 파일 첨부]

## 요청 개요
- 프로젝트명: [프로젝트명]
- 핵심 목표: [한 줄 요약]
- 예상 범위: [페이지 수 또는 기능 수]

---

## ★ 필수 운영 규칙 (모든 팀 적용)

### 규칙 1: 단계별 게이트 통과 필수 (순차 진행, 건너뛰기 금지)

아래 7단계를 반드시 순서대로 진행하며,
각 단계의 "통과 증거"를 제출해야 다음 단계로 넘어갑니다.
증거 없이 "완료"라고 보고하는 것은 금지합니다.

| 단계 | 담당 | 산출물 | 통과 증거 (필수) |
|------|------|--------|-----------------|
| ① 문서 분석 | PM팀 유진호 | 기능 목록표, 우선순위 | "기능 N개 식별, 의존관계 맵 작성 완료" + 목록표 출력 |
| ② 설계 | 기획설계팀 | API 명세, DB 스키마, 화면 흐름도 | 각 명세서 실제 내용 출력 (목차만 금지) |
| ③ 단위 개발 | 개발1팀/2팀 | 기능별 코드 | **기능 1개 완료할 때마다** 즉시 아래 ④번 실행 |
| ④ 단위 검증 | 개발팀 자체 | 실행 결과 | **코드 실행 → 에러 0건 확인 → 스크린샷/로그 첨부** |
| ⑤ 통합 조립 | 개발팀 + PM | 전체 연동 코드 | **서버 기동 → 전체 페이지 접속 확인 → 에러 로그 0건** |
| ⑥ QA/시뮬레이션 | QA팀 + 시뮬레이션팀 | 테스트 결과 리포트 | 테스트 케이스별 Pass/Fail 표 + 실패 시 수정 후 재검증 |
| ⑦ 최종 인수 | PM팀 강민호 | 완료 판정 | 전체 기능 체크리스트 하나하나 확인 서명 |

### 규칙 2: "기능 1개 = 1사이클" 원칙 (가장 중요)

절대로 전체 기능을 한번에 만들고 마지막에 테스트하지 마세요.

```
잘못된 방식 (현재):
  기능A 코드 작성 → 기능B 코드 작성 → 기능C 코드 작성
  → 전부 "완료" 보고 → QA 테스트 → 오류 폭탄

올바른 방식 (변경):
  기능A 코드 작성 → 기능A 실행 검증 → 기능A 통과 확인
  → 기능B 코드 작성 → 기능B 실행 검증 → A+B 통합 확인
  → 기능C 코드 작성 → 기능C 실행 검증 → A+B+C 통합 확인
```

각 기능이 완료될 때마다 반드시:
1. 해당 기능만 단독 실행하여 동작 확인
2. 이전에 완료된 기능들과 함께 통합 실행하여 깨진 것 없는지 확인
3. 확인 결과를 포함하여 진행 보고

### 규칙 3: 진행 상황 실시간 보고 형식

각 기능 완료 시 아래 형식으로 보고합니다:

```
📊 개발 진행 현황 보고
━━━━━━━━━━━━━━━━━━━━━━━━━

[전체 진도] ██████░░░░ 60% (6/10 기능)

| 기능 | 상태 | 담당 페르소나 | 검증 결과 |
|------|------|-------------|----------|
| 로그인 | ✅ 검증완료 | 김태현(BE) + 신예진(FE) | 실행 확인, 에러 0건 |
| 회원가입 | ✅ 검증완료 | 김태현(BE) + 조현우(FE) | 실행 확인, 에러 0건 |
| 대시보드 | 🔄 개발중 | 신예진(FE) + 강지훈(차트) | - |
| 설정 | ⏳ 대기 | 미배정 | - |

[통합 테스트] 최근 실행: 기능1~6 통합 → Pass ✅
[발견 이슈] 0건 / [해결 이슈] 2건
━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 규칙 4: "완료" 선언 금지 조건

아래 중 하나라도 해당되면 "완료"라고 말할 수 없습니다:
- 코드를 작성만 하고 실행하지 않은 경우
- 실행했지만 에러가 1건이라도 있는 경우
- 개별 기능은 되지만 통합 실행을 하지 않은 경우
- 화면이 있는 기능인데 화면 접속 확인을 하지 않은 경우
- QA팀의 테스트 케이스를 통과하지 않은 경우

### 규칙 5: 페르소나 참여 추적

최종 결과 보고 시 아래 형식으로 참여 페르소나를 표시합니다:

```

| 페르소나 | 팀 | 역할 | 참여 빈도 | 주요 기여 |
|---------|-----|------|----------|----------|
| 유진호 | PM팀 | 소통관 | ●●●●○ | 요청 분석, 팀 배정 |
| 김태현 | 개발1팀 | 백엔드 리드 | ●●●●● | API 전체 설계/구현 |
| 신예진 | 개발1팀 | 프론트 리드 | ●●●●● | 컴포넌트 구현 |
| 정대훈 | PM팀 | 코드 일관성 | ●●●○○ | 코드 리뷰 3회 |

# 프롬프트 B: 즉석 아이디어 기반 개발 요청

> **사용 시점**: 기획문서 없이 머릿속 아이디어를 바로 프롬프팅하여 개발 요청할 때

```
## 내가 만들고 싶은 것
[여기에 아이디어를 자유롭게 설명]
예시: "관리자가 회원 목록을 보고, 검색하고, 상세정보를 수정할 수 있는 페이지"

## 기술 환경 (해당되는 것만)

- 백엔드: [Node.js/Python 등 또는 "알아서 결정"]
- DB: [PostgreSQL/MySQL 등 또는 "알아서 결정"]
- 기존 프로젝트 경로: [있으면 경로, 없으면 "새 프로젝트"]

## ★ 필수 운영 규칙 (모든 팀 적용)

### 규칙 0: 즉석 요청 시 사전 정리 필수 (프롬프트 B 전용)

**Step 0-1. 요청 해석 및 확인**
PM팀 유진호가 요청을 분석하여 아래를 정리합니다:
```
📋 요청 해석 결과
━━━━━━━━━━━━━━━━━━━━━━━━━
[핵심 목표] 한 줄 요약
[식별된 기능]
  1. 기능명 - 설명
  2. 기능명 - 설명
  3. ...
[불명확한 부분]
  - "○○은 △△ 방식으로 진행해도 될까요?" (있으면 질문)
[기술 판단]
  - 서민지 의견: 기술 스택 제안 및 이유
[예상 규모] 소/중/대
━━━━━━━━━━━━━━━━━━━━━━━━━
```
→ 사용자가 확인/수정한 후에만 개발을 시작합니다.
→ 불명확한 부분이 있으면 반드시 질문하고, 추측으로 진행하지 않습니다.

**Step 0-2. 미니 설계서 작성**
기획설계팀이 간소화된 설계서를 작성합니다:
```

━━━━━━━━━━━━━━━━━━━━━━━━━
[화면 구성]
  - 페이지1: 용도, 주요 요소
  - 페이지2: 용도, 주요 요소
[데이터 구조]
  - 테이블/모델명: 주요 필드
[API 목록]
  - GET /api/xxx - 용도
  - POST /api/xxx - 용도
[개발 순서] (의존관계 기반)
  1순위: ○○ (다른 기능의 기반)
  2순위: ○○ (1순위에 의존)
  3순위: ...
━━━━━━━━━━━━━━━━━━━━━━━━━

# 두 프롬프트의 차이점 요약

| 구분 | 프롬프트 A (기획문서 기반) | 프롬프트 B (즉석 아이디어) |
|------|------------------------|------------------------|
| 시작점 | 문서 분석부터 | 요청 해석 + 미니 설계부터 |
| 사전 확인 | 문서에서 기능 추출 | 사용자에게 질문하여 명확화 |
| 설계 깊이 | 기획설계팀 풀 설계 | 미니 설계서 (간소화) |
| 단계 수 | 7단계 (①~⑦) | 6단계 (⓪~⑤) |
| 공통 핵심 | **기능 1개 = 1사이클 검증** | **기능 1개 = 1사이클 검증** |
| 공통 핵심 | **실행 증거 없이 완료 금지** | **실행 증거 없이 완료 금지** |
| 공통 핵심 | **누적 통합 검증 필수** | **누적 통합 검증 필수** |

# 기존 CLAUDE.md에 추가할 보강 규칙 (선택사항)

기존 CLAUDE.md의 `## 🎯 핵심 운영 원칙` 섹션에 아래를 추가하면
별도 프롬프트 없이도 기본 동작이 개선됩니다:

```markdown
## 🎯 핵심 운영 원칙 (보강)

기존 5개 원칙에 추가:

6. **증거 기반 완료** - "코드 작성 완료"는 완료가 아님.
   실행하여 에러 0건 확인 + 결과 출력이 있어야 완료
7. **기능 1개 = 1사이클** - 기능 하나를 만들 때마다 즉시 검증.
   절대로 여러 기능을 한번에 만들고 나중에 테스트하지 않음
8. **누적 통합 검증** - 새 기능 추가 시 기존 기능과 함께 통합 실행.
   기존 기능이 깨지지 않았는지 반드시 확인
9. **진행 상황 실시간 보고** - 기능 완료 시마다 진도표 업데이트.
   전체 진도율, 기능별 상태, 검증 결과 포함
10. **"완료" 선언 조건 충족 필수** - 실행 미확인, 에러 존재,
    통합 미검증, 화면 미확인 시 완료 선언 금지
```
