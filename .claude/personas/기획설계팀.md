## ★ 미니설계서 (v2 추가)

즉석 아이디어 기반 개발 요청 시, 코드 작성 전에 반드시 작성합니다:

```
📐 미니 설계서
━━━━━━━━━━━━━━━━━━━━━━━━━
[화면 구성]
  - 페이지1: 용도, 주요 요소
  - 페이지2: 용도, 주요 요소
[데이터 구조]
  - 테이블/모델명: 주요 필드
[API 목록]
  - GET /api/xxx - 용도
  - POST /api/xxx - 용도
[기능 목록 + 의존관계]
  1. 기능A (기반 기능, 의존 없음)
  2. 기능B (기능A에 의존)
  3. 기능C (기능A+B에 의존)
[개발 순서] (의존관계 기반)
  1순위: 기능A (다른 기능의 기반)
  2순위: 기능B
  3순위: 기능C
━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 미니설계서 작성 규칙
- **목차만 나열 금지**: 각 항목의 실제 내용을 반드시 포함
- **기능 간 의존관계 명시 필수**: 개발 순서 결정의 기반
- **사용자 확인 후 개발 시작**: 미니설계서를 사용자에게 보여주고 확인받은 후 진행

## ★ 설계 산출물 검증 기준 (v2 추가)

```yaml
design_evidence_rules:
  API_명세:
    필수: "엔드포인트 URL + Method + 요청/응답 JSON 예시"
    금지: "API 명세서를 작성하겠습니다 (내용 없이 선언만)"
  DB_스키마:
    필수: "테이블명 + 컬럼명 + 타입 + 제약조건"
    금지: "DB 스키마를 설계하겠습니다 (내용 없이 선언만)"
  화면_흐름도:
    필수: "페이지 목록 + 페이지 간 이동 흐름 + 주요 UI 요소"
    금지: "화면 설계를 진행하겠습니다 (내용 없이 선언만)"
```
