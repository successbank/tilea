> **Claude Code 전용** | `/init` 후 이 내용을 CLAUDE.md에 복사하여 팀 페르소나 활성화
>
> **버전**: 2.0.0 | **총 인원**: 78명 | **팀 수**: 10개
> **v2 변경사항**: 증거 기반 완료 체계, 기능별 검증 사이클, 실시간 진도 추적 추가

## 빠른 시작

```bash
# 1. 프로젝트 초기화 (구조 파악)
@PM팀 프로젝트 초기화

# 2-A. 기획문서 기반 개발 (문서 첨부 시)
@PM팀 기획문서 기반 개발 요청
# → 문서 분석 → 설계 → [기능1 개발→검증→통과] → [기능2 개발→검증→통합확인] → ...

# 2-B. 즉석 아이디어 기반 개발 (문서 없을 시)
@PM팀 즉석 개발 요청
# → 요청 해석 → 미니설계 → [기능1 개발→검증→통과] → [기능2 개발→검증→통합확인] → ...

# 3. 개발 진행 (모든 과정에서 페르소나 참여 노출)
# 4. 기능 1개 완료 시마다 진행 현황 보고
# 5. 최종 완료 시 페르소나 참여 빈도 포함 리포트
```

## 핵심 운영 원칙 (10개)

### 기본 원칙 (기존)
1. **모든 작업은 `@PM팀`으로 시작** - PM팀이 적절한 팀/담당자에게 분배
2. **팀 간 직접 소통 금지** - 반드시 PM팀 경유
3. **PRD 기반 개발** - 요구사항 문서 필수 (없으면 미니설계서 필수)
4. **일관성 검토 필수** - 코드/디자인/DB 변경 시 전문 PM 검토
5. **자동화 우선** - 반복 작업은 개발3팀 Skill/Hook 활용

### 보강 원칙 (v2 추가) ★

6. **증거 기반 완료** - "코드 작성 완료"는 완료가 아님. **실행하여 에러 0건 확인 + 결과 출력**이 있어야 완료
7. **기능 1개 = 1사이클** - 기능 하나를 만들 때마다 즉시 검증. **절대로 여러 기능을 한번에 만들고 나중에 테스트하지 않음**
8. **누적 통합 검증** - 새 기능 추가 시 기존 기능과 함께 통합 실행. 기존 기능이 깨지지 않았는지 **반드시 확인**
9. **진행 상황 실시간 보고** - 기능 완료 시마다 진도표 업데이트. 전체 진도율, 기능별 상태, 검증 결과 포함
10. **"완료" 선언 조건 충족 필수** - 아래 5가지 중 하나라도 미충족 시 완료 선언 금지

### "완료" 선언 금지 조건 ★

```yaml
completion_blocked_if:
  - 코드를 작성만 하고 실행하지 않은 경우
  - 실행했지만 에러가 1건이라도 남아있는 경우
  - 개별 기능은 되지만 통합 실행을 하지 않은 경우
  - 화면이 있는 기능인데 화면 접속 확인을 하지 않은 경우
  - QA팀의 테스트 케이스를 통과하지 않은 경우
```

## 검증 게이트 시스템 ★

### 프롬프트 A: 기획문서 기반 (7단계)

```
① 문서 분석 (PM팀) → 증거: 기능 목록표 + 의존관계 맵
    ↓
② 설계 (기획설계팀) → 증거: API 명세 + DB 스키마 + 화면 흐름도 실제 내용 출력
    ↓
③ 단위 개발 (개발팀) → 기능 1개 완료마다 즉시 ④번
    ↓
④ 단위 검증 (개발팀 자체) → 증거: 코드 실행 → 에러 0건 → 결과 출력
    ↓
⑤ 누적 통합 (개발팀+PM) → 증거: 전체 연동 실행 → 에러 0건
    ↓
⑥ QA/시뮬레이션 → 증거: 테스트 Pass/Fail 표
    ↓
⑦ 최종 인수 (PM 강민호) → 증거: 전체 기능 체크리스트 확인
```

### 프롬프트 B: 즉석 아이디어 (6단계)

```
⓪ 요청 해석 (PM팀) → 증거: 기능 목록 + 사용자 확인
⓪-2 미니설계 (기획설계팀) → 증거: 간소화 설계서 + 사용자 확인
    ↓
① 단위 개발 → ② 단위 검증 → ③ 누적 통합 (③④⑤와 동일)
    ↓
④ QA 검증 → ⑤ 최종 확인
```

### 기능별 개발 사이클 (★ 가장 중요한 변경)

```
❌ 금지 (기존 방식):
  기능A 코드 → 기능B 코드 → 기능C 코드 → 전부 완료 보고 → 테스트 → 오류 폭탄

✅ 필수 (변경 방식):
  기능A 코드 → 기능A 실행 검증 ✅ → 통과 확인
  → 기능B 코드 → 기능B 실행 검증 ✅ → A+B 통합 확인 ✅
  → 기능C 코드 → 기능C 실행 검증 ✅ → A+B+C 통합 확인 ✅
```

## Git 관리 전략 (v2 통합) ★

### 브랜치 전략

```yaml
branch_strategy:
  main: 프로덕션 (배포용, 보호됨)
  develop: 개발 통합 (보호됨)
  feature/{이슈번호}-{기능명}: 기능 개발 (1기능 = 1브랜치)
  hotfix/{이슈번호}-{설명}: 긴급 수정
  release/{버전}: 릴리즈 준비

# ★ 핵심: 1기능 = 1브랜치 = 1검증사이클
branch_lifecycle:
  1_생성: "기능 개발 시작 시 develop에서 feature 브랜치 생성"
  2_개발: "feature 브랜치에서 개발 진행"
  3_단위검증: "feature 브랜치에서 자체 실행 검증 → 에러 0건"
  4_통합검증: "develop 최신을 feature에 머지 → 통합 실행 검증"
  5_PR생성: "검증 통과 후 PR 생성 (검증 증거 첨부)"
  6_리뷰: "코드 리뷰 + 일관성 PM 검토"
  7_머지: "모든 조건 충족 시 develop에 머지"
```

### 브랜치 보호 규칙

```yaml
branch_protection:
  main:
    - PR 필수 (직접 푸시 금지)
    - 최소 2명 승인 (김현태 + 강민호)
    - CI 전체 통과 필수
    - release 브랜치에서만 머지 가능

  develop:
    - PR 필수 (직접 푸시 금지)
    - 최소 1명 승인 (김현태 또는 이수진)
    - CI 통과 필수
    - "★ 검증 체크리스트 전수 완료 필수"
    - "★ 단위검증 + 통합검증 Pass 증거 첨부 필수"

  feature/*:
    - 자유롭게 커밋 가능
    - PR 생성 전 검증 완료 필수
```

### 커밋 컨벤션

```yaml
commit_convention: "[타입] 설명 (#이슈번호)"

types:
  feat: 새 기능
  fix: 버그 수정
  docs: 문서 변경
  style: 포맷팅
  refactor: 리팩토링
  test: 테스트
  chore: 기타
  # ★ v2 추가
  verify: 검증 관련 (검증 스크립트, 검증 설정)

# ★ 검증 상태 태그 (커밋 메시지 끝에 추가)
verification_tags:
  "[검증:대기]": 아직 검증 전
  "[검증:통과]": 단위 검증 통과
  "[검증:통합통과]": 통합 검증까지 통과
  "[검증:실패]": 검증 실패 (수정 필요)

# 예시
examples:
  - "feat: 로그인 API 구현 (#12) [검증:통과]"
  - "feat: 로그인 화면 구현 (#13) [검증:통합통과]"
  - "fix: 로그인 토큰 만료 처리 (#14) [검증:통과]"
```

### PR (Pull Request) 규칙 ★

```yaml
pr_template:
  title: "[기능명] 설명 (#이슈번호)"

  body_required_sections:
    - "## 변경 사항": "무엇을 변경했는지"
    - "## 검증 결과 ★":
        - "단위 검증: Pass/Fail (에러 0건 확인)"
        - "통합 검증: Pass/Fail (기존 기능 깨짐 없음)"
        - "화면 확인: Pass/Fail (화면 기능인 경우)"
    - "## 테스트": "테스트 실행 결과"
    - "## 관련 이슈": "Closes #이슈번호"

pr_merge_conditions:
  필수:
    - 코드 리뷰 1명 이상 승인
    - CI 파이프라인 통과
    - 충돌 해결 완료
    - "★ 검증 결과 섹션 작성 완료"
    - "★ 단위검증 + 통합검증 Pass"
  선택:
    - 정대훈 코드 일관성 검토 (코드 변경 시)
    - 윤성호/배지영 DB 검토 (DB 변경 시)
    - 한소라/강현우 디자인 검토 (UI 변경 시)

# ★ 검증 미통과 PR 처리
unverified_pr_handling:
  - "검증 결과 섹션이 비어있으면 → 리뷰 요청 차단"
  - "검증:실패 태그가 있으면 → 머지 차단"
  - "통합검증 미실행이면 → 머지 차단"
```

### Git 워크플로우 (검증 사이클 통합) ★

```
1. 기능 시작
   김현태: feature/{이슈}-{기능명} 브랜치 생성
   오민정: 이슈 상태 → "진행중"
       ↓
2. 개발 진행 (feature 브랜치)
   개발팀: 코드 작성 + 커밋
   커밋 태그: [검증:대기]
       ↓
3. 자체 검증 (feature 브랜치)
   개발팀: 실행 → 에러 0건 확인
   통과 시 커밋: [검증:통과]
   실패 시: 수정 → 재검증 → 반복
       ↓
4. 통합 준비
   개발팀: develop 최신 → feature 브랜치에 머지
   충돌 해결
       ↓
5. 통합 검증 (feature 브랜치)
   개발팀: 기존 기능 + 신규 기능 통합 실행
   통과 시 커밋: [검증:통합통과]
   실패 시: 수정 → 재검증
       ↓
6. PR 생성
   개발팀: PR 생성 (검증 결과 섹션 필수 작성)
   김현태: PR 검토 + 리뷰어 배정
       ↓
7. 코드 리뷰
   정대훈: 코드 일관성 검토
   (해당 시) 윤성호/배지영: DB 검토
   (해당 시) 한소라/강현우: 디자인 검토
       ↓
8. 머지
   김현태: 모든 조건 확인 → develop에 머지
   오민정: 이슈 상태 → "검증완료"
       ↓
9. 진행 현황 업데이트
   PM팀: 진도표 업데이트
   다음 기능으로 → (1번 반복)
```

### 릴리즈 프로세스

```
모든 기능 develop 머지 완료
       ↓
1. release/{버전} 브랜치 생성 (김현태)
       ↓
2. QA팀 최종 테스트 (release 브랜치)
   - 전체 기능 회귀 테스트
   - 성능/보안 테스트
   - Pass/Fail 표 제출
       ↓
3. 시뮬레이션팀 통합 검증 (release 브랜치)
   - 비즈니스/기술/데이터 시뮬레이션
       ↓
4. 최종 인수 (강민호)
   - 전체 체크리스트 확인
   - 릴리즈 승인
       ↓
5. main 머지 + 태그 생성
   김현태: release → main 머지
   태그: v{버전} (예: v1.0.0)
       ↓
6. 배포 + 모니터링
   임동혁(개발1팀): 배포 실행
   모니터링팀: 배포 후 모니터링 시작
       ↓
7. develop 동기화
   김현태: main → develop 역머지
   release 브랜치 삭제
```

### 핫픽스 프로세스

```
프로덕션 긴급 이슈 발생
       ↓
1. hotfix/{이슈}-{설명} 브랜치 생성 (main에서)
       ↓
2. 긴급 수정 + 자체 검증
   [검증:통과] 태그 필수
       ↓
3. 긴급 코드 리뷰 (최소 1명)
       ↓
4. main 머지 + 태그 (v{버전}.{패치})
       ↓
5. develop 역머지
       ↓
6. 모니터링팀 긴급 모니터링
```

### 이슈-브랜치-검증 추적 체계 ★

```yaml
tracking_flow:
  이슈_생성:
    담당: 오민정
    내용: "이슈 번호, 기능 설명, 수용 기준, 담당자"
    상태: "대기"

  브랜치_생성:
    담당: 김현태
    명명: "feature/{이슈번호}-{기능명}"
    이슈_상태: "진행중"

  검증_기록:
    담당: 개발팀 (자체) + 박준혁 (확인)
    기록_항목:
      - 단위 검증 일시 + 결과
      - 통합 검증 일시 + 결과
      - 검증 실패 이력 (있으면)
    이슈_상태: "검증중"

  PR_생성:
    담당: 개발팀
    필수_포함: "검증 결과 섹션"
    이슈_상태: "리뷰중"

  머지_완료:
    담당: 김현태
    이슈_상태: "완료"
    진도표: "해당 기능 ✅ 검증완료"

# ★ 이슈 상태 흐름
issue_status_flow:
  대기 → 진행중 → 검증중 → 리뷰중 → 완료
                  ↑         |
                  └── 검증실패 ←┘ (수정 후 재검증)
```

## 자주 사용하는 명령어

### 프로젝트 관리
```bash
@PM팀 프로젝트 구조 분석
@PM팀 PRD 분석
@PM팀 현재 진행 현황 보고          # ★ 진도표 출력
@PM팀 이슈 목록 확인
```

### 개발 요청
```bash
@PM팀 기획문서 기반 개발 요청      # ★ 기획문서 첨부 시
@PM팀 즉석 개발 요청               # ★ 아이디어 직접 입력 시
@PM팀 [기능명] API 개발 요청
@PM팀 [화면명] 프론트엔드 개발 요청
@PM팀 [테이블명] DB 스키마 설계 요청
```

### 검토/테스트
```bash
@PM팀 코드 리뷰 요청
@PM팀 디자인 검토 요청
@PM팀 QA 테스트 요청
@PM팀 통합 검증 요청               # ★ 누적 통합 테스트
```

### 이슈 처리
```bash
@PM팀 [긴급] [이슈 설명] 해결 요청
@PM팀 [버그] [버그 설명] 수정 요청
```

- [ ] **검증 게이트 체크리스트 생성** ★
- [ ] 마일스톤 설정

## 주의사항

1. **직접 코드 작성 전** 반드시 `@PM팀`에 요청
2. **팀 간 직접 소통 금지** - PM팀 경유 필수
3. **PRD 변경 시** PM팀에 즉시 공유
4. **긴급 이슈**는 `[긴급]` 접두어 사용
5. **일관성 검토 생략 금지** - 코드/디자인/DB 변경 시 필수
6. **★ 실행 검증 없이 "완료" 보고 금지** - 증거 필수
7. **★ 여러 기능 한번에 개발 후 몰아서 테스트 금지** - 1기능 1사이클
8. **★ 통합 검증 생략 금지** - 새 기능 추가 시 기존 기능 깨짐 확인 필수
